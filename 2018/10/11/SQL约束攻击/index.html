SQL约束攻击

发表于 2018-10-11 | 分类于 web |

呃……做题遇到了一个叫做约束攻击的攻击方法，就深入了解了一下 ，貌似蛮有趣。

part one

mysql> desc liuyanban;

+———-+——————+——+—–+———————+—————-+

| Field | Type | Null | Key | Default | Extra |

+———-+——————+——+—–+———————+—————-+

| id | int(10) unsigned | NO | PRI | NULL | auto_increment |

| username | varchar(100) | YES | | NULL | |

| password | varchar(50) | YES | | NULL | |

| content | varchar(200) | YES | | NULL | |

| datetime | datetime | NO | | 0000-00-00 00:00:00 | |

+———-+——————+——+—–+———————+—————-+

5 rows in set (0.01 sec)

可以看到id为该表的主键，主键具有唯一性，username字段没设置为主键或者UNIQUE约束，那就会造成可以以任意身份登陆。那这就说明我们可以从username下手约束攻击了。
在执行sql语句时候，字符串末尾的空格会被删除，也就是’wang’和’wang ‘是一样的。注意，对尾部空白符的这种修剪操作，主要是在字符串比较期间进行的。mysql在储存的时候会默认把空格删除，原因是在执行where等需要字符串比较的时候，会使用空格填充字符串，长度保持一致后再进行对比。

$query = “SELECT *FROM users WHERE username=’wang’”;
$query = “SELECT *FROM users WHERE username=’wang ’”;
在本地cmd里使用了一下，两个查询语句结果一样。
那么可以使用wang登陆了！

ps：当然也有例外，在LIKE子句中，是不会删除空格的。

part two

还有就是每个字段都会定义一个长度，例如char(5)，所以在执行Insert时，超过这个长度的都会被截断。

“dawang”会被截断成“dawan”……
存进数据库的便是dawan
那么如果别人用dawan注册了一个账号，就可以登陆dawan了。
